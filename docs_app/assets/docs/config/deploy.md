# `.flow_deploy.json`

Source of truth for `flow deploy`. Generated by `flow deploy init`; hand-edit
to fill in credentials and changelogs.

The file lives at the project root.

## Top-level fields

### `skip_version_increment`

Type: `bool` · Default `true`.

When `true`, the build pipeline does not modify `pubspec.yaml` during a
deploy. Override per-command with `--increment-version` to bump the patch +
build number for a single run.

### `profiles`

Type: `Map<String, ProfileConfig>` · Optional.

Named build/deploy presets. Each key (e.g. `dev`, `staging`, `production`)
becomes invokable as `flow <profile>` or `flow deploy run <profile>`. See
[Profiles](#profiles-1) below.

### `build`

Type: `Object` · Optional.

Top-level defaults that apply when no profile is selected and no CLI flag
overrides them.

- `flavor` — passed to Flutter as `--flavor`.
- `target` — passed to Flutter as `--target`.

When you use profiles, prefer setting these inside each profile rather than at
the top level.

## `android.*`

### `android.package_name`

Type: `String` · Recommended for flavored builds.

Tells Fastlane (`supply`) which Google Play listing to upload to. **Must
match** the `applicationId` Gradle produces for the flavor you're building.

:::info Where to get this
Open `android/app/build.gradle.kts` (or `build.gradle`). For a non-flavored
app, it's the top-level `applicationId`. For a flavored app, it's
`<base>.<flavorName>` if `use_suffix` is `true`.
:::

### `android.json_key_path`

Type: `String` · Required for Google Play uploads.

Filesystem path (relative to project root) to a Google Play service-account
JSON key with **Release Manager** permissions.

:::info Where to get this
1. Open the [Google Play Console](https://play.google.com/console).
2. **Setup** → **API access**.
3. Either link an existing Google Cloud project or create a new one.
4. Under **Service accounts**, click **Create new service account**.
5. Follow the prompts to Google Cloud, generate a JSON key, download it.
6. Back in Play Console → **API access**, grant the new service account the
   **Release Manager** role.
7. Save the JSON file somewhere in the repo (e.g. `secrets/`) and **never
   commit it** — add to `.gitignore`.
:::

### `android.changelog`

Type: `Map<String, String>` · Required for `flow deploy update`.

Localized release notes. Keys are BCP-47 locale codes that Play Console
recognizes; values are the changelog text.

```json
"changelog": {
  "en-US": "Bug fixes and improvements",
  "ar-SA": "إصلاحات وتحسينات",
  "fr-FR": "Corrections de bugs et améliorations"
}
```

### `android.firebase_app_distribution`

Type: `Object` · Required for `flow deploy beta --provider firebase` on Android.

#### `app_id`

Type: `String` · Required.

Firebase **App ID** — the long identifier under Project Settings → General →
"Your apps" → Android app → **App ID**. Format:
`1:<sender-id>:android:<random-hex>`.

:::info Where to get this
1. Open the [Firebase Console](https://console.firebase.google.com).
2. Pick your project.
3. ⚙️ → **Project settings** → **General**.
4. Scroll to "Your apps", select the Android app for this flavor.
5. Copy **App ID**. Do **not** confuse this with the Android `applicationId`.
:::

#### `groups`

Type: `String` · Optional (recommended).

Comma-separated tester group aliases configured in Firebase App Distribution.

:::info Where to get this
Firebase Console → **App Distribution** → **Testers & Groups** → create
groups (e.g. `qa-team`) and add testers to each. The "alias" column is the
value you put here.
:::

#### `testers`

Type: `String` · Optional.

Comma-separated email addresses, in addition to `groups`. Useful for one-off
invitees.

#### `release_notes`

Type: `String` · Optional.

The note that appears in the tester's invitation email. Falls back to the
first non-empty `android.changelog` entry if omitted.

## `ios.*`

### `ios.app_identifier`

Type: `String` · Recommended for flavored builds.

iOS bundle identifier. Must match the Xcode `PRODUCT_BUNDLE_IDENTIFIER` for
the scheme you're building.

:::info Where to get this
Open `ios/Runner.xcodeproj/project.pbxproj` and search for
`PRODUCT_BUNDLE_IDENTIFIER`, or in Xcode select the **Runner** target → the
**Signing & Capabilities** tab → the **Bundle Identifier** field. For
flavored apps with `use_suffix: true`, the non-production identifiers append
`.<flavor>` to the base.
:::

### `ios.app_store_connect`

Type: `Object` · Required for iOS uploads.

App Store Connect API credentials. All three fields are required.

#### `key_id`

Type: `String` · Required.

The 10-character key identifier shown next to your generated API key.

#### `issuer_id`

Type: `String` · Required.

The Issuer ID for your team — a UUID. Same across all keys in your team.

#### `key_filepath`

Type: `String` · Required.

Path (relative to project root) to the downloaded `.p8` private key file.

:::info Where to get all three
1. Open [App Store Connect](https://appstoreconnect.apple.com).
2. **Users and Access** → **Keys** (under "Integrations").
3. Click **+** to create a new key.
4. Give it a name and grant it the **App Manager** role.
5. **Generate**.
6. Copy the **Issuer ID** displayed at the top of the page (use as `issuer_id`).
7. Note the **Key ID** in the new row (use as `key_id`).
8. Click **Download API Key** — Apple will only let you download the `.p8`
   **once**. Save it to e.g. `secrets/AuthKey_<key_id>.p8` and add to
   `.gitignore`.
:::

:::danger Treat the .p8 like a password
Anyone with the `.p8` can publish builds to App Store Connect as your team.
Store it in a CI secret manager, not in the repository.
:::

### `ios.changelog`

Type: `Map<String, String>` · Required for `flow deploy update`.

Localized "What's New" copy submitted to App Store Connect for the new
version. Keys are App Store Connect locale codes
(`en-US`, `ar-SA`, `fr-FR`, etc.).

### `ios.testflight`

Type: `Object` · Optional.

Controls how Fastlane's `pilot` action uploads the iOS build to TestFlight.

#### `enable_external_testing`

Type: `bool` · Default `false`.

- `false` — upload becomes available to **internal** testers only (App Store
  Connect users on your team). No review required. The other `testflight.*`
  fields can be left blank.
- `true` — upload becomes available to **external** testers (anyone with a
  TestFlight invite link). Apple requires beta app review information; the
  other `testflight.*` fields become required.

#### `groups`

Type: `String` · Required when `enable_external_testing` is `true`.

Comma-separated TestFlight group names. Create them in App Store Connect →
your app → **TestFlight** → **External Testing** → **Add Group**.

#### `beta_app_feedback_email`

Type: `String` · Required when `enable_external_testing` is `true`.

Public-facing email shown in the TestFlight beta invitation.

#### `beta_app_review_info`

Type: `Object` · Required when `enable_external_testing` is `true`.

Contact information for the App Review team. Mostly self-explanatory; the
demo account fields are conditional.

| Field | Required when |
|---|---|
| `contact_email`, `contact_first_name`, `contact_last_name`, `contact_phone` | always (when external testing enabled) |
| `demo_account_required` | always |
| `demo_account_name`, `demo_account_password` | only when `demo_account_required` is `true` |
| `notes` | optional free-form notes shown to reviewers |

### `ios.firebase_app_distribution`

Same shape as `android.firebase_app_distribution`, but for the iOS Firebase app.
The iOS App ID format is `1:<sender-id>:ios:<random-hex>`.

## Profiles

### `profiles.<name>`

Each profile is a preset that pre-configures one deploy invocation. Profile
names become invokable as:

```bash
flow staging                  # shortcut
flow deploy run staging       # explicit form
```

#### `mode`

Type: `String` · Required.

One of `beta` or `update`. Determines whether the build goes through the beta
or update pipeline.

#### `provider`

Type: `String` · Required.

One of `fastlane`, `firebase`, `mixed`.

- `fastlane` — uploads via Fastlane to TestFlight / App Store / Play Store.
- `firebase` — uploads via Firebase CLI to App Distribution.
- `mixed` — Android via Firebase, iOS via Fastlane. Requires `platform: all`.

#### `platform`

Type: `String` · Required.

`ios`, `android`, or `all`.

#### `build`

Type: `Object` · Optional.

Per-profile `flavor` / `target` overrides for `flutter build`.

#### `android.package_name` / `ios.app_identifier`

Type: `String` · Optional.

Per-profile overrides — useful when a single project has multiple bundle
identifiers (e.g. a staging build that points to a different App Store
Connect record).

## Validation

`flow` validates `.flow_deploy.json` on every command. Missing required
credentials trigger a fail-fast error before any build runs:

```terminal
$ flow deploy beta -p ios
[E] Missing ios.app_store_connect.key_id in .flow_deploy.json
[E] Missing ios.app_store_connect.issuer_id in .flow_deploy.json
[E] Missing ios.app_store_connect.key_filepath in .flow_deploy.json
exit 1
```

:::warn Keep secrets out of git
`.flow_deploy.json` itself is safe to commit — but the files it points at
(`AuthKey_*.p8`, the Google Play service account JSON) are credentials. Add
the secrets directory to `.gitignore` and load real values from a secret
store in CI.
:::
