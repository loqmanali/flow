# `flow flavor reset`

Restores the project to its original, **non-flavored** state. Useful when
flavors were a mistake, or when you want to re-init from scratch.

## Synopsis

```bash
flow flavor reset
```

## What it removes

- `.flow_flavor.json`
- Every `lib/main_<flavor>.dart` entry point
- The generated `AppConfig` Dart file at `<app_config_path>`
- Generated scripts under `scripts/` (if `generate_scripts: true` was set)
- All per-flavor `ios/Flutter/<flavor>.xcconfig` files
- Per-flavor Xcode build configurations and schemes
- All flavor entries from `.vscode/launch.json`
- Firebase per-flavor `firebase_options_<flavor>.dart` files (if Firebase
  was wired)

## What it reverts

- `android/app/build.gradle` (or `.kts`) â€” `flavorDimensions` and
  `productFlavors` blocks are removed; the original single-flavor structure
  is restored.
- `ios/Runner.xcodeproj` â€” base build configurations are restored, schemes
  are pruned to the default `Runner` scheme.

## When to use this

- You set up flavors with the wrong base package ID and want to start over.
- You're migrating an app away from flavors (e.g. consolidating to a single
  staging/production CI variable instead).
- A failed `init` or `replace` left the project in an inconsistent state and
  you want a clean slate.

## What it does **not** touch

- Your application code (anything not generated by `flow`).
- `pubspec.yaml`.
- App Store Connect / Google Play / Firebase records â€” these are external
  systems and `flow` never reaches them.

:::danger Irreversible
Unlike `replace`, `reset` does not take a snapshot. Once you confirm, the
deleted files are gone. If you've made hand-edits to the generated
`AppConfig`, copy them out first.

If `.flow_flavor.json` is still under version control, you can recover by
restoring it and running `flow flavor init --from .flow_flavor.json`.
:::
